# Coding Harness Containers

This document defines the expected contract for containerized coding harnesses in this repository. A coding harness is a Dockerized CLI agent or coding assistant definition such as `cecli`, `claudecode`, or `opencode`.

## Goals

A harness container should make an agent usable from any Linux or macOS host with predictable behavior:

- run against a mounted repository without requiring host-global installs;
- bridge host/project configuration into the container;
- preserve enough repository structure for monorepos and git-aware tools;
- keep secrets out of images while making runtime credentials available;
- provide a consistent build/run/test/debug surface for each definition.

## Definition Contract

Each harness definition should expose, where applicable:

- `Dockerfile` — image build definition.
- `entrypoint.sh` — runtime setup, config discovery, env bridging, and final CLI launch.
- `build.sh` — definition-local image build wrapper.
- `run.sh` — definition-local docker run wrapper.
- `test.sh` and/or `tests/` — smoke and contract checks.
- `README.md` — usage, image names, mounts, config behavior, and harness-specific notes.
- `debug.sh` — optional troubleshooting shell when the harness needs it.
- Sample config files such as `sample_opencode.json` or `.env.sample`.

The README should explicitly mark whether the definition is experimental, candidate, or stable.

## Runtime Configuration Discovery

Harnesses should load configuration in this order unless the underlying tool requires a stricter precedence:

1. **Project-local config** mounted in the working tree.
2. **User home config** under `$HOME` inside the container.
3. **Baked defaults** copied into image-level locations such as `/opt/<harness>/...`.
4. **Common macOS/Linux secondary locations**, when useful:
 - `/opt/<harness>/...`
 - `/usr/local/share/<harness>/...`
 - `/opt/homebrew/share/<harness>/...`
 - `$HOME/Library/Application Support/<harness>/...` for macOS-compatible layouts.
5. **Minimal inline fallback** generated by `entrypoint.sh` only when no config file exists.

Do not assume a host `$HOME` or host `/opt` is mounted. Runtime discovery happens inside the container.

## Environment Bridging

Harnesses should source a project `.env` file when present, using `set -a` so loaded values are exported to the child CLI. `.env` files must be runtime-only; never bake secrets into the image.

Entrypoints should bridge common project aliases into tool-specific env vars. Examples:

- Provider keys:
 - `ANTHROPIC_API_KEY`
 - `OPENAI_API_KEY`
 - `OPENROUTER_API_KEY`
 - `XAI_API_KEY`
 - `GEMINI_API_KEY` / `GOOGLE_API_KEY`
 - `DEEPSEEK_API_KEY`
 - `GROQ_API_KEY`
 - `MISTRAL_API_KEY`
- Model aliases:
 - `ARCHITECT_MODEL` — preferred planning/primary model when the harness has only one main model slot.
 - `EDITOR_MODEL` — preferred edit/small/secondary model when the harness exposes such a slot.
 - `SMALL_MODEL` — fallback lightweight model.
 - Tool-specific overrides such as `OPENCODE_MODEL`, `OPENCODE_SMALL_MODEL`, and `CECLI_MODEL` should not be overwritten if already set.
- UI aliases:
 - `DARK_MODE`, `CODE_THEME`, and harness-specific equivalents.

When a tool requires provider-prefixed model IDs, normalize common bare names where practical, while preserving explicit `provider/model` values.

## Config Files to Bridge

Harnesses should detect and/or seed tool-specific config files. Examples include:

- opencode:
 - `opencode.json`, `opencode.jsonc`
 - `.opencode/agents/*.md`
 - `AGENTS.md`
 - global `~/.config/opencode/opencode.json`
 - global `~/.config/opencode/agents/*.md`

 - `.env`
 - conventions files used by the harness prompt
- cecli:
 - `.cecli.config.yml`
 - `.cecliignore`
 - `.env`
 - runtime home/config directories
- Claude Code-style harnesses:
 - project instructions such as `CLAUDE.md` or `AGENTS.md`;
 - project `.claude/` folders mounted from the working tree;
 - home `.claude/` folders discovered under `$HOME/.claude` inside the container;
 - tool/MCP configuration files;
 - auth/config directories that must be mounted or seeded.

Harness-specific config should be documented in that harness README, including where defaults are baked and how to force reseeding.

## Monorepo Support

Harnesses should handle monorepos without flattening repository structure.

When a user starts from a subproject, the wrapper should preserve the repo-relative path under `/app`, mount root-level git state, and expose only the intended target when the tool supports ignore files.

Expected behavior:

- Detect the repository root with `git rev-parse --show-toplevel` when available.
- Detect the current repo-relative scope with `git rev-parse --show-prefix`.
- Mount the selected subfolder at `/app/<relative-scope>` rather than always flattening it to `/app`.
- Mount root `.git` into `/app/.git` when the harness needs git history, repo maps, or diff context.
- Preserve root-level files required by the tool or package manager, such as:
 - workspace manifests: `pnpm-workspace.yaml`, root `package.json`, `turbo.json`, `nx.json`;
 - lockfiles: `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, `poetry.lock`;
 - shared config: `tsconfig*.json`, `.npmrc`, `.yarnrc.yml`, `.tool-versions`;
 - agent instructions: `AGENTS.md`, `CONVENTIONS.md`, `CLAUDE.md`;
 - tool config: `.cecli.config.yml`, `.cecliignore`, `opencode.json`.

For **`.env`**: do not treat it as a file to preserve into the agent mount set when
credential isolation matters. In `firewall` mode the target posture is host-exported keys
(or host-side resolution into `broker.env` on the egress sidecar only). Wrappers
may overlay a symlink-resolved `.env` at `/app/.env` for entrypoint autoload convenience
(`internal/workspace`); that is not a security control. In `broker` mode, prefer host
`export` / `docker run -e` over bind-mounting `.env` when minimizing file exposure.

- Make the chosen scope visible in the container name or startup preamble so parallel harness sessions are distinguishable.



## Dependency Bootstrapping

Harnesses may install project dependencies on first run, but should be conservative:

- only install when a project manifest exists and dependencies are missing;
- prefer lockfile-respecting commands such as `pnpm install` or `npm ci`;
- do not modify lockfiles unless the underlying task explicitly requires it;
- make automatic dependency installation visible in startup logs;
- allow disabling install behavior when the harness supports read-only or audit workflows.

## Security and Secrets

Harness images should:

- run as a non-root user by default;
- avoid setuid/setgid binaries unless explicitly required;
- avoid baking credentials, auth files, or `.env` contents into the image;
- source secrets only at runtime from mounted files or `docker run -e` values;
- keep session/config persistence under proveo-owned `~/.proveo` (default-on via
  `harness.manifest` `home:`); never bind-mount host IDE homes (`~/.cursor`, `~/.claude`);
- scrub deny-listed auth filenames from proveo home each run; keep inference auth on
  env / credential-broker keys;
- avoid logging secret values while still reporting whether required keys were detected.

**Credential isolation by egress mode** (see [`README.md`](README.md) and
[`_spec/paradigms.md`](_spec/paradigms.md)):

| Mode | Secret confinement |
| --- | --- |
| `firewall` | Real secrets only in `broker.env` on the egress sidecar. Agent `.env` is masked; `load_env` skips; secret env is a sentinel (`proveo-brokered`). |
| `proxy` | No TLS inspection — broker-style inject/strip is impossible. Squid limits destinations; agent holds credentials in-process (or withholds them, requiring another auth path). |
| `broker` | No egress boundary — dev-only; `.env` may be mounted for convenience; accept in-process secrets. |

The credential broker is **firewall-mode only**. Do not document or implement broker injection
for `proxy`/`broker` without an explicit topology change (adding TLS MITM to those modes).

## Observability and Debuggability

Entrypoints should print a concise preamble before launching the tool:

- harness/tool version;
- runtime versions such as node/npm/pnpm when relevant;
- detected project config files;
- detected agent/instruction files;
- whether provider keys or project config are missing;
- mounted working directory and monorepo scope when available.

A `debug.sh` wrapper is recommended when troubleshooting requires shell access with the same mounts/env behavior as `run.sh`.

## Testing Expectations

Harness tests should verify the contract, not just that the image builds:

- image builds successfully;
- expected CLI/runtime tools are installed;
- container runs as the intended non-root user;
- baked defaults exist;
- entrypoint seeds or preserves config correctly;
- forced reseeding works if supported;
- `.env` loading and env alias bridging work without printing secrets;
- monorepo mount behavior is covered by smoke tests when wrappers implement it;
- live provider/API tests are skipped unless the required key is present.
